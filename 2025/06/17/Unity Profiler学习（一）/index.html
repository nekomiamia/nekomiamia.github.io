目的

感觉需要深入学习一下关于Unity性能优化相关的东西，那就需要先分析游戏的性能，于是开始学习Unity Profiler，这篇也是边学边写，就当是记录并分享一下学习过程吧。

Unity Profiler

是什么？（扒的官方文档）

Unity 性能分析器 (Unity Profiler) 是一种可以用来获取应用程序性能信息的工具。可以将性能分析器连接到网络中的设备或连接到已连接到计算机的设备，从而测试应用程序在目标发布平台上的运行情况。还可以在 Editor 中运行性能分析器，从而在开发应用程序时概要了解资源分配情况。

性能分析器可以收集并显示有关应用程序各个方面（例如 CPU、内存、渲染器和音频）的性能数据。该工具可用于识别应用程序中可提高性能的方面，并在这些方面进行迭代。可以查明代码、资源、场景设置、摄像机渲染和构建设置如何影响应用程序性能等方面的情况。性能分析器在一系列图表中显示结果，因此可以直观地查看应用程序性能出现尖峰的位置。

CPU Usage

CPU Usage Profiler 模块的图表可跟踪应用程序主线程中花费的时间。

图表类别 描述
Rendering 应用程序花费多少时间来渲染图形。
Scripts 应用程序花费多少时间来运行脚本。
Physics 应用程序在物理引擎上花费多少时间。
动画 应用程序花费多少时间来动画化应用程序中带蒙皮的网格渲染器 (Skinned Mesh Renderers)、游戏对象和其他组件。这还包括针对 Animation 组件和 Animator 组件所用的系统进行计算所花费的时间。
GarbageCollector 应用程序花费多少时间来运行垃圾回收器。
VSync 应用程序每帧花费多少时间来等待 targetFrameRate 或下一个要同步的 VBlank。此时间基于 QualitySettings.vSyncCount 值、目标帧率或者 VSync 设置（即运行应用程序的平台的默认或强制最大值）。有关 VSync 的更多信息，请参阅本文档中的渲染和 VSync 样本部分。
Global Illumination 应用程序在光照中花费多少时间。
UI 应用程序花费多少时间来显示其 UI。
Others 应用程序在不属于任何其他类别的代码中花费多少时间。此事件包括整个 EditorLoop 或者是 Editor 中对运行模式进行性能分析时的性能分析开销等方面。

1. Scripts

  • 为什么重要：几乎所有游戏逻辑（AI、输入、状态更新）都在脚本里执行，过高表示逻辑瓶颈或不合理的循环/大量分配。
  • 优化建议：通过 Hierarchy 查看 Self 时间高的函数，减少不必要的 Update 调用或频繁 GC.Alloc。

2. Rendering

  • 为什么重要：代表 CPU 在渲染命令（剔除、Draw Call 准备等）上的开销，高则说明渲染路径瓶颈。
  • 优化建议：检查批次（batches）与 SetPass 调用，减少切换、合并静态网格、使用 GPU Instancing。

3. GarbageCollector

  • 为什么重要：GC 运行时会卡帧，导致帧率不稳定，高频 GC.Alloc 是罪魁祸首。docs.unity3d.com
  • 优化建议：尽量避免运行时分配，使用对象池或 struct，并监测 GC.Alloc 列数据。

4. VSync / WaitForTargetFPS

  • 为什么重要：如果 CPU 时间低于 targetFrameRate，系统会主动等待（VSync），这段时间表现为 “等待” 而非瓶颈。
  • 优化建议：注意这个指标来判断是否真正存在性能压力，或只是高帧空闲。

GPU Usage

图表类别 描述
Opaque 内置渲染管线渲染不透明对象的时间。
Transparent 内置渲染管线渲染透明对象的时间。
Shadows/Depth 内置渲染管线渲染阴影贴图的时间。
Deferred PrePass 内置延迟渲染管线渲染深度缓冲区预通道 (pre pass) 的时间。
Deferred Lighting 内置延迟渲染管线处理光照的时间。
PostProcess 内置渲染管线处理后期处理效果的时间。
Other 处理可编程渲染管线等其他事务的渲染时间。

Memory

类别 描述
Total Allocated 应用程序已使用的内存总量。
Texture Memory 应用程序中的纹理已使用的内存量。
Mesh Memory 应用程序中的网格已使用的内存量。
Material Count 应用程序中的材质实例数量。
Object Count 应用程序中的原生对象实例数量。
GC Used Memory GC 堆使用的内存量。
GC Allocated in Frame GC 堆中每帧分配的内存量。

OKOK，这个估计是Profiler中最重要的部分了，GC，很多面试官问Profiler其实是想引导你说出GC，所以了解GC其实很重要的。

先说一下GC Used Memory和GC Allocated in Frame这两个指标吧。

GC Used Memory

  • 定义：当前托管堆（managed heap）实际使用的内存量，即已分配但尚未被回收的内存。
  • 用途：
    • 显示应用当前持有多少 GC 堆资源；
    • 持续增长可能表明内存泄漏或对象未及时释放；
    • 间接反映 GC 的触发频率与负载变化。

GC Allocated in Frame

  • 定义：每一帧中新分配的托管堆内存量，来自 GC.Alloc 的累积（以字节计）。
  • 用途：
    • 用于监测每帧分配波动；
    • 突然上升可能意味着运行时大量产生临时对象，如 string 拼接或 foreach 中隐式分配（例如迭代器对象）。
    • 帧内分配越多，意味着 GC 可能更频繁触发，导致卡顿。

一言以蔽之，GC Used Memory是内存的总用量， GC Allocated in Frame是每帧新分配的内存量。

GC又是什么？GC就是垃圾回收，垃圾回收 是一种 自动内存管理机制，用于回收程序中 不再被引用 的对象所占的堆内存，从而避免内存泄漏与程序崩溃。简单来说，就是当托管堆空间不足的时候，就会触发GC，从入口开始标记所有可达的对象，那不可达的就意味着没引用了呗，就是垃圾需要被回收掉。

  1. 托管堆：托管堆是由 CLR（.NET）或 Mono/IL2CPP（Unity）在运行时分配的一块连续内存区域，专用于存储所有 引用类型 的对象。当执行 new MyClass() 时，系统会在托管堆中分配该对象并返回一个引用，而非直接指针

  2. GC的起点：垃圾回收是需要一个入口的，在 C#（包括 Unity 中的托管环境）及 .NET 垃圾回收机制中，GC 从一组称为 Roots（根对象） 的集合为起点，识别整个对象图中哪些对象仍被引用。

    常见的Roots有：

    • 静态字段和常量：类的 static 变量以及 const／readonly 常量，会一直保留引用，成为根节点之一
    • 局部变量跟方法参数：正在执行的方法栈帧中，活跃的局部变量和参数引用的对象，也被视为 GC roots

  3. .NET Vs Unity：.NET的GC是分代的，也就是分成了0 ~ 2三代，每次都是从年轻代开始GC，存活的就升级到更高一代，只有回收完当代还空间不足，才往上一级GC。Unity的GC是不分代的，每次都要扫一遍堆，所以堆越大GC越慢，而且它是Stop-the-world式的GC，在执行GC的时候会暂停整个程序，造成卡顿，后续虽然引入了增量GC将GC过程分散到多个时间片，但仍有卡顿。

GC的优化

  1. 避免在 Update 等高频调用中分配对象，使用结构体（struct）替代类，字符串拼接改用StringBuilder，避免多次新建字符串对象。
  2. 使用对象池进行对象的复用，避免频繁生成销毁GameObject。对必要的组件进行缓存，避免多次GetComponent。
